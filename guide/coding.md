# ZZ的代码风格Guide

* 规则是用于帮助提升可读性、可维护性和保持一致性的工具，而不是目的
* 项目应当使用lint和类型检查工具（对于动态类型语言），并开启警告
  - 例外：在项目早期可以暂时忽略，但应当在之后补全
    * 举例：[Galosity](https://github.com/HZZcode/Galosity)在早期使用JavaScript，后来增加了ESLint并且迁移到TypeScript
* 在必要时可以在局部禁用lint规则，但禁用域应该尽可能小；在原因并不明显时应当使用注释说明，但在修改代码时必须更新注释；此条对警告同样适用
  - 举例：[Galosity:utils/logger.ts](https://github.com/HZZcode/Galosity/blob/main/ts/utils/logger.ts)在单行禁用了ESLint的no-console规则
* 性能优化必须以profiling作为依据
* 代码应当自解释，注释只应当作为文档和用于说明并不显然的实现方案
  - 举例：[TheWitnessSolver/DefaultedDict.py](https://github.com/HZZcode/TheWitnessSolver/blob/master/DefaultedDict.py)用注释说明了为什么需要`.copy()`
* 只在必要的情况下进行类型标注
  - 举例：可以自动推导出类型的变量不应该标注类型（在强类型语言中用`var`或是`auto`一类的机制让编译器推导类型）
    * 例外：C++的`auto`有各种问题（比如无符号数、proxy类、引用规则，等等），所以还是需要视情况而定
* 在创建项目时应当使用最新的语言、工具链和库版本，并且在合理的时候无需顾虑使用新版本特性
  - 举例：ZZ的网页服务器使用ESNext语言版本，并用TypeScript转译和CoreJS的Polyfill保持浏览器兼容性
* 当标准库功能够用时不要自己造轮子
  - 举例：C#中使用`Action<>`和`Func<>`而不是手写`delegate`
* 上传到GitHub的项目最好使用GitHub Actions以自动化构建和代码风格检查
* 代码文件应当使用UTF-8编码
* 缩进应当是4个空格；空行不应该有空格
* 当且仅当元素和之后的括号不在同一行时使用尾随逗号
* 二元运算符前后应该有空格，而一元运算符后则不应有
  - 例外：Python的规范中，对于没有类型标注的默认参数，`=`前后不应有空格，此时以语言规范为准
* 在一行代码太长时应当进行换行；换行时应当从操作符（例如，赋值号`=`、链式调用的`.`、三元运算符中的`?`和`:`，等等）的前面断开，并且增加一个缩进；大括号换行遵循K&R风格
  - 例外：对于长url或者命令等不换行时更方便的特殊情况可以不换行
  - 举例：
  ```ts
  function f() {
      const a_really_long_named_result_list
          = a_really_long_named_list
              .filter(a_really_long_named_predicate)
              .map(a_really_long_named_function);
      // ...
  }
  ```
  - 例外：在使用formatter或语言有一般惯例时可以使用其默认配置
    - 举例：C#中一般惯例是在大括号前换行
* 命名应当是可读的英文单词/短语，并且在项目内保持一致；对于同样的含义，在同一个项目中应该使用同样的单词；在不损害可读性的前提下应当尽可能简洁
  - 例外：作用域充分短或者有特殊惯例的情况下，可以使用单字母名称
    * 举例：整数循环变量通常使用`i`,`j`,`k`作为名称
  - 例外：在抄写数学公式时可以忽略命名规范直接使用原记号
  - 举例：如果语言没有内置的访问器机制，永远应该使用get/set作为命名
  - 举例：命名不应该包含类型信息，因为任何现代IDE都有悬停显示功能
* 一般地，类/类型名称使用UpperCamel，函数/变量使用lowerCamel，文件名使用kebab-case或者snake_case
  - 例外：如果使用的语言有特殊规定或者惯例，以语言规定/惯例为准
    * 举例：在C++/Rust中，函数/变量使用snake_case；在Java/C#中，文件名和类名同样使用UpperCamel；C#中public字段/函数也使用UpperCamel
* 未使用的变量应当命名为纯下划线
  - 例外：在Python中，如果个别情况下需要通过参数名调用函数，可以使用其它方法避免警告
* 对于与语言关键词重名的变量，可以在后面加上下划线区分
  - 例外：如果语言提供了内置的机制，使用该机制
    * 举例：C#中可以在变量名前加上`@`以规避重名
* 导入语句应当有固定的顺序，且通常应当位于文件开头
  - 举例：使用ESLint规则`simpleImportSort`对`import`排序
  - 例外：如果某个导入仅在函数局部使用，可以在函数开头导入（这一点常用于Python的`from pprint import pp`）；动态导入不受此限制
* 在异常作为语言原生机制的语言中，在合适时可以使用，并尽可能使用语言已经给出的异常；异常只应该用于不正常的情况
  - 举例：[BishRuntime/BishObject.cs](https://github.com/labbish/Bish/blob/master/BishRuntime/BishObject.cs)中许多可能失败的操作提供返回可空值的`TryDoSomething`和会抛出异常的`DoSomething`版本，后者由前者实现（只需要一个`??`运算符，这很简单）
* `try`-`catch`的作用域和捕获的异常范围应当尽可能小；在忽略异常时，如果原因并不明显应当进行说明
* 自定义异常应当有特殊后缀
  - 举例：视语言命名规范通常可以为`Exception`或`Error`
* 对于弃用的API应当予以标注；在将来需要改进的地方使用`TODO`注释，尚未实现或者有问题的地方使用`FIXME`注释；尽可能使用语言提供的机制进行这些标注
  - 举例：在Rust中未实现的功能使用`todo!`宏
* 变量的作用域应当尽可能小；在不需要修改时声明为`const`，并且尽可能在定义时初始化
  - 举例：全局变量通常都不是一个好的模式：可以使用类的static字段替代，或者重构以消除全局状态
* 在不影响语义和可读性的情况下，应当尽可能减少块缩进
  - 举例：使用`if`-guard提前退出函数；在C#中`using var`和`namespace`都应使用单行形式
  - 举例：单个表达式的箭头函数不应有大括号：任何现代IDE都能一键将它们重构成`=> { return result; }`形式，所以不需要担心这样带来的重构问题
* 不应重复默认行为
  - 举例：在JavaScript中，默认的空构造函数是不必要的
* `switch`-`case`的每个`case`都不应当越过分支向下执行，除非它们自身是空的；`case`应当是穷尽的
  - 例外：如果你可以确定穷尽性而编译器不能，在`default`分支中抛出异常，并在错误信息表明原因
* 不应比较布尔值和`true`或`false`
* 对于长段重复性逻辑，应当使用函数封装，但不应过度抽象；函数不应该过长和完成太多任务；函数重载应当清晰，并且参数的默认值应当是常量
  - 举例：Python中，应当用`None`作为可变类型参数的默认值，然后在函数体内判断
* 内部不变性应当通过私有属性和公开方法维护
* 可以使用局部函数来封装内部逻辑和减少函数参数
  - 举例：[TheWitnessSolver/Path.py](https://github.com/HZZcode/TheWitnessSolver/blob/master/Path.py)中，使用局部函数`finder`实现深度优先搜索，同时避免显式传递不变的参数；在其它DFS场景中也可参考这个模式
* 公开API应当进行参数校验
  - 例外：参数校验不需要和typing重复
  - 例外：如果可以确定参数校验是性能热点，则可以提供不带校验的版本，并加以说明
    * 举例：在Rust中不带校验的版本通常应标注为`unsafe`
* 应当尽可能使用声明式的生成器或链式调用代替手写循环
* 在遍历容器时，如果不需要索引，应当使用`for`-`in`式的写法；如果需要索引，在语言支持的情况下应当尽可能使用`for`-`in`同时迭代索引和元素
  - 举例：在python中使用`enumerator`函数同时迭代索引和元素
* 在语言支持时，应当使用getter和setter而非getXxx和setXxx函数；没有特殊需求的成员变量则应直接声明为`public`而不是增加额外的访问器
  - 例外：很遗憾，Java在这一点上对于重构并不友好，需要视情况而定
* 对于字符串格式化应当尽可能使用内插字符串，在有本地化等需求时使用format-string
* 对于单双引号都用于字符串字面量的语言，应当统一使用单引号
  - 例外：字符串中包含单引号时，为了减少转义可以使用双引号
* 对于外部资源，保证释放时机的确定性
  - 对于C++/Rust，使用RAII模式在析构函数/`drop`中释放；对于GC语言，使用它们内置的`with`-block之类的机制
* 对于使用构造函数的语言，如果需要异步地构造、返回单例或其它难以实现的情况，应当将构造函数声明为私有并使用工厂方法
  - 举例：[BishRuntime/BishInt.cs](https://github.com/labbish/Bish/blob/master/BishRuntime/BishInt.cs)中使用工厂方法实现小整数的常量池
* 对于支持类继承的语言，继承关系应当尽可能明显；特别地，在支持的语言中应当避免不合理的多继承
  - 举例：C++的`iostream`是典型的反例：应当组合`Reader`和`Writer`实现读写
* 对于支持类继承的语言，复写关系应当是显式的
  - 举例：在Java中使用`@Overrides`；C#中视具体需求使用`override`或`new`；TypeScript中使用`override`（这样也能更好地保持重构代码时的正确性）
* 在支持运算符重载的语言中，应当保持重载运算符的语言明显
  - 举例：在C++中，通常不应该重载`&&`和`||`，因为它们具有短路语义
